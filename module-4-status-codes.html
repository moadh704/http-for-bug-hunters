Status Codes & Behavior

HTTP status codes are not just success/failure signals. They often act as powerful oracles during testing and can leak significant information about the application.

Learning Objectives

  • Understand what different status code classes reveal in practice
  • Learn how to use status codes as testing oracles
  • Identify information leakage through error responses
  • Recognize inconsistent behavior between users/roles

Status Code Classes & Security Relevance

2xx Success

200, 201, 204 are the most common. Inconsistent responses between different privilege levels often reveal access control issues.

3xx Redirection

Open redirect vulnerabilities frequently live here. Also check whether sensitive headers (cookies, Authorization) are sent to third-party domains.

4xx Client Errors

400, 401, 403, 404, 422. Differences in responses (timing, content, status) between valid and invalid users can confirm the existence of resources or bypass authorization.

5xx Server Errors

These are goldmines. Stack traces, internal paths, framework versions, and database errors are frequently leaked.

Practical Testing Techniques

  • Compare responses for the same request as different users/roles.
  • Send malformed or unexpected input and carefully analyze error messages.
  • Use timing differences as oracles (e.g., 404 vs 403 response times).
  • Never ignore 5xx responses — they often reveal internal implementation details.

Key Takeaways

  • Status codes can act as powerful information oracles.
  • Inconsistent behavior between users is often a sign of access control problems.
  • Error responses frequently leak sensitive internal information.
  • Always analyze both the status code and the full response body/headers.
