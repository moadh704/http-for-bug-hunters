HTTP Methods Deep Dive

HTTP methods define the intent of a request. Misunderstanding or misusing them is one of the most common sources of authorization bypasses and business logic vulnerabilities.

Learning Objectives

  • Understand the semantic meaning and security implications of each HTTP method
  • Know the difference between safe, idempotent, and unsafe methods
  • Learn how to test method-related vulnerabilities effectively
  • Identify when applications accept unexpected methods

Understanding HTTP Methods

HTTP methods indicate the desired action to be performed on a resource. The most important ones for security testing are:

GET

Retrieves data. Should be safe (no side effects) and idempotent (repeating it produces the same result).

Common Vulnerabilities:

  • IDOR by changing IDs in the URL
  • Sensitive data exposure in query parameters
  • SSRF when the application fetches URLs from GET parameters

POST

Submits data to create or trigger actions. Not idempotent by default.

Common Vulnerabilities:

  • CSRF on state-changing actions
  • Mass assignment (adding unexpected fields)
  • Business logic flaws (price manipulation, workflow bypass)
  • Lack of rate limiting on sensitive endpoints

PUT

Replaces an entire resource. Should be idempotent.

Common Vulnerabilities:

  • Weak or missing ownership checks
  • Mass assignment of fields the UI never exposes
  • Overwriting resources belonging to other users

PATCH

Partial update to a resource. Often implemented later and validated less strictly than PUT.

Common Vulnerabilities:

  • Mass assignment
  • Weaker authorization than equivalent PUT endpoints
  • Logic flaws in partial updates

DELETE

Removes a resource. High-risk method.

Common Vulnerabilities:

  • Missing authorization / ownership checks
  • Ability to delete other users' resources
  • Soft delete bypasses

Security Testing Methodology

When testing any endpoint, always ask these questions:

  • What methods does the endpoint actually accept?
  • Can I bypass controls by using a different method?
  • Does the application properly enforce authorization for each method?
  • Are there undocumented methods that work?

Pro Tip: Use tools like Burp Suite's Repeater or ffuf to quickly test different methods against the same endpoint.
