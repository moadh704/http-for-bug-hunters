HTTP Methods Deep Dive

Understanding HTTP methods is fundamental to finding many authorization and business logic vulnerabilities.

Learning Objectives

  • Know the purpose and expected behavior of common HTTP methods
  • Understand which methods are idempotent and why it matters
  • Learn how to test methods for security issues
  • Identify when applications misuse HTTP methods

Core HTTP Methods

GET

Used to retrieve resources. Should be safe and idempotent. Parameters usually go in the URL.

Common Issues: IDOR via ID manipulation, sensitive data in query parameters, SSRF through URL parameters.

POST

Used to submit data or trigger actions. Not idempotent by default.

Common Issues: CSRF, mass assignment, lack of rate limiting, business logic flaws.

PUT

Used to replace a resource entirely. Should be idempotent.

Common Issues: Missing authorization checks, mass assignment of fields.

PATCH

Used for partial updates. Often implemented later and less strictly validated.

Common Issues: Mass assignment, weaker authorization than PUT endpoints.

DELETE

Used to remove resources. High impact if authorization is weak.

Common Issues: Horizontal/vertical privilege escalation through missing ownership checks.

Security Testing Approach

For every endpoint you test, ask:

  • What method is documented vs what actually works?
  • Can I use a different method to bypass controls?
  • Does the method behave idempotently as expected?
  • Are there undocumented methods accepted?

Key Takeaways

  • Never assume only documented methods are accepted.
  • PUT and PATCH are often less protected than POST.
  • DELETE endpoints frequently have weak authorization.
  • Testing method swapping is a quick and high-value technique.
